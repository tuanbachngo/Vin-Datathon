# Question for Datathon organizers — Part 3 (Sales Forecasting)

## Context
The task brief says *"Forecast the Revenue column over the time period in `sales_test.csv`"*, and each row is the tuple `(Date, Revenue, COGS)` for 2023-01-01 → 2024-07-01.

The provided `sample_submission.csv` already contains non-trivial values in both `Revenue` and `COGS` columns. We checked:

- `COGS / Revenue` ratio distribution in the sample submission (IQR 0.81 – 0.91) matches train_2022 almost exactly (IQR 0.79 – 0.92).
- Mean daily COGS in the sample submission (2,783,810) matches train_2022 (2,795,672) to within 0.4 %.
- `corr(Revenue, COGS) = 0.984` inside the sample submission — consistent with train's 0.976.

This is either (a) real, organizer-provided future COGS used as an exogenous feature, or (b) a plausibility-matched placeholder generated by a baseline.

## Question

Please confirm one of the following:

1. **The `COGS` column in `sample_submission.csv` is the true test-window COGS and may be used as an input feature to the forecasting model.**
2. **`COGS` in `sample_submission.csv` is a placeholder only — participants must predict (or ignore) COGS, and it must not be used as a feature.**

## Why it matters

- Under (1), `Revenue ≈ COGS / expected_cogs_ratio` is a very strong modeling prior — it changes the feature set, the model architecture, and the achievable accuracy ceiling.
- Under (2), the model must forecast `Revenue` from calendar + historical-lag features only, since no other auxiliary table in the provided data covers the 2023-2024 test window.

## Secondary questions (only if (1) is true)

- Is `COGS` graded alongside `Revenue`, or is only `Revenue` evaluated?
- If `COGS` is also graded, should participants submit a predicted `COGS` or echo the given values verbatim?
